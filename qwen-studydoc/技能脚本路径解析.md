# Pi 编码助手技能脚本路径解析分析

## 执行摘要

本文档分析了 pi 编码助手如何解析技能中引用的脚本，识别路径解析问题的根本原因，并提供了正确的脚本位置处理解决方案。

## 1. 架构概述

### 1.1 技能系统组件

pi 编码助手的技能系统由以下三个核心组件组成：

1. **技能发现** (`src/core/skills.ts`)
   - 扫描目录中的 `SKILL.md` 文件
   - 从 YAML frontmatter 中提取元数据（名称、描述）
   - 根据 Agent Skills 规范进行验证
   - 返回包含 `baseDir` 和 `filePath` 的 `Skill` 对象

2. **技能注册** (`src/modes/interactive/interactive-mode.ts`)
   - 将技能转换为斜杠命令 (`/skill:name`)
   - 存储映射：`commandName -> skill.filePath`
   - 注册到自动补全提供者

3. **技能扩展** (`src/core/agent-session.ts`)
   - 将 `/skill:name args` 扩展为完整的技能内容
   - 将技能指令注入到 agent 上下文中
   - 提供 `baseDir` 引用用于相对路径

### 1.2 技能数据结构

```typescript
interface Skill {
  name: string;              // 来自 frontmatter 或目录名
  description: string;       // 必需，用于发现
  filePath: string;          // SKILL.md 的绝对路径
  baseDir: string;           // SKILL.md 的父目录
  source: string;            // "user"、"project" 或 "path"
  disableModelInvocation: boolean;
}
```

## 2. 脚本解析流程

### 2.1 当前实现

当通过 `/skill:name args` 调用技能时：

1. **命令解析** (`agent-session.ts:883` 的 `_expandSkillCommand`)
   ```typescript
   private _expandSkillCommand(text: string): string {
     if (!text.startsWith("/skill:")) return text;
     
     const spaceIndex = text.indexOf(" ");
     const skillName = spaceIndex === -1 ? text.slice(7) : text.slice(7, spaceIndex);
     const args = spaceIndex === -1 ? "" : text.slice(spaceIndex + 1).trim();
     
     const skill = this.resourceLoader.getSkills().skills.find((s) => s.name === skillName);
     if (!skill) return text;
     
     try {
       const content = readFileSync(skill.filePath, "utf-8");
       const body = stripFrontmatter(content).trim();
       const skillBlock = `<skill name="${skill.name}" location="${skill.filePath}">\nReferences are relative to ${skill.baseDir}.\n\n${body}\n</skill>`;
       return args ? `${skillBlock}\n\n${args}` : skillBlock;
     } catch (err) {
       // 错误处理...
       return text;
     }
   }
   ```

2. **技能块注入**
   - 读取完整的 `SKILL.md` 内容
   - 去除 YAML frontmatter
   - 包装在 `<skill>` XML 块中
   - 注入 `baseDir` 引用："References are relative to {baseDir}"
   - 在技能内容后追加用户参数

3. **Agent 处理**
   - 将扩展的技能块作为用户消息接收
   - Agent LLM 读取技能指令
   - LLM 使用工具执行任务（包括 `bash` 工具）

### 2.2 脚本引用模式

技能可以通过多种方式引用脚本：

## 3. 根本原因分析

### 3.1 问题描述

**问题**：技能中引用的脚本使用相对路径，但 bash 执行的工作目录没有正确设置为技能的 `baseDir`。

**当前行为**：
1. 技能加载时 `baseDir = "/repo/.pi/skills/myskill"`
2. 技能内容引用 `scripts/myscript.py`
3. Agent 执行 bash 命令：`python scripts/myscript.py`
4. **问题**：bash 在 `process.cwd()`（项目根目录）执行，而不是 `baseDir`
5. 结果：找不到 `scripts/myscript.py`

### 3.2 路径解析不匹配

不匹配发生在两个层面：

**层面 1：技能加载（正确）**
```typescript
// skills.ts:259
return {
  skill: {
    name,
    description: frontmatter.description,
    filePath,              // d:\repo\.pi\skills\myskill\SKILL.md
    baseDir: skillDir,     // d:\repo\.pi\skills\myskill
    source,
    disableModelInvocation: frontmatter["disable-model-invocation"] === true,
  },
  // ...
}
```

**层面 2：Bash 执行（不正确）**
```typescript
// bash.ts:188
export function createBashTool(cwd: string, options?: BashToolOptions): AgentTool {
  // cwd 从 agent 会话传递，通常是 process.cwd()
  // 不是技能的 baseDir！
  
  return {
    execute: async (_toolCallId, { command, timeout }, signal, onUpdate) => {
      // 命令在 'cwd' 上下文中执行
      // 如果技能引用 './scripts/myscript.py'，它会在 cwd 而不是 baseDir 中查找
    }
  }
}
```

### 3.3 问题原因

1. **Bash 工具设计**：`bash` 工具接受 `cwd` 参数，但技能没有指定它
2. **Agent 上下文**：技能作为文本注入，而不是具有自己上下文的可执行单元
3. **LLM 行为**：LLM 读取技能指令，但不知道正确的 working directory
4. **缺少技能上下文**：技能的 `baseDir` 在提示中被提及，但没有被强制执行

## 4. 解决方案

### 4.1 解决方案 1：技能感知 Bash 工具（推荐）

修改 bash 工具以接受技能上下文并自动设置工作目录。

**实现**：

```typescript
// 为 bash 工具执行添加技能上下文
interface BashToolOptions {
  // ... 现有选项
  skillBaseDir?: string;  // 新增：技能的基础目录
}

function createBashTool(cwd: string, options?: BashToolOptions): AgentTool {
  const skillBaseDir = options?.skillBaseDir;
  
  return {
    execute: async (_toolCallId, { command, timeout }, signal, onUpdate) => {
      // 确定有效的工作目录
      let effectiveCwd = cwd;
      
      // 如果技能上下文存在且命令是相对路径，使用技能的 baseDir
      if (skillBaseDir && !isAbsolute(command)) {
        // 检查命令是否以 ./ 或 ../ 开头或不包含 /
        if (command.startsWith('./') || command.startsWith('../') || !command.includes('/')) {
          effectiveCwd = skillBaseDir;
        }
      }
      
      // 使用解析的 cwd 执行
      return new Promise((resolve, reject) => {
        spawn(shell, [...args, command], {
          cwd: effectiveCwd,
          // ... 其余执行逻辑
        });
      });
    }
  }
}
```

**优点**：
- 对技能透明
- 与现有技能格式兼容
- 无破坏性更改

**缺点**：
- 需要修改 bash 工具执行逻辑
- 可能影响非技能 bash 命令

### 4.3 解决方案 3：技能专用 Bash 工具包装器

**技能中的用法**：

private _expandSkillCommand(text: string): string {
  if (!text.startsWith("/skill:")) return text;
  
  const spaceIndex = text.indexOf(" ");
  const skillName = spaceIndex === -1 ? text.slice(7) : text.slice(7, spaceIndex);
  const args = spaceIndex === -1 ? "" : text.slice(spaceIndex + 1).trim();
  
  const skill = this.resourceLoader.getSkills().skills.find((s) => s.name === skillName);
  if (!skill) return text;
  
  try {
    const content = readFileSync(skill.filePath, "utf-8");
    const body = stripFrontmatter(content).trim();
    const skillBlock = `<skill name="${skill.name}" location="${skill.filePath}">\nReferences are relative to ${skill.baseDir}.\n\n${body}\n</skill>`;
    return args ? `${skillBlock}\n\n${args}` : skillBlock;
  } catch (err) {
    // 错误处理...
    return text;
  }
}

## 10. 结论

核心问题是技能使用相对路径引用脚本，但 bash 执行上下文不会自动切换到技能的 `baseDir`。LLM 收到关于路径解析的说明，但没有被强制执行。

**立即修复**：使用方案 4（提示工程）在技能块中显式说明路径解析规则。

**长期修复**：实现提示增强加上 bash 工具中的智能默认检测的混合方法。

**关键见解**：技能应该是具有自己执行上下文的自包含单元。当前实现将技能视为文本注入，但它们应该被视为具有自己工作目录的微型项目。
